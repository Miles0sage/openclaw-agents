What Are Leaked Prompts?

How the best AI systems really work, decoded from publicly leaked source code.

What's a System Prompt?

Every AI model has a "system prompt" — invisible instructions that shape how it behaves. Think of it like firmware for a robot. The system prompt tells Claude:

  • • What your identity is (you're an expert developer)
  • • What you must ALWAYS do (read before editing)
  • • What you must NEVER do (delete without confirmation)
  • • How to make decisions (cost-aware, quality-first)
  • • What tools you have access to

Why Do Companies Keep Them Secret?

Because system prompts are the moat. They're not the LLM (that's commodity now — GPT-4, Opus, Kimi are all comparable). The moat is the instructions that make the LLM actually useful in production. So companies guard them jealously.

But over the past few years, engineers have leaked them:

  • • Devin (GitHub security researcher)
  • • Claude Code (Anthropic team members)
  • • Manus (GitHub gist)
  • • Cursor (source code scrape)

The 8 Patterns That Make AI Work in Production

1. Identity Anchoring

Place your identity at the TOP of the prompt. "You are Claude, an expert developer. You have X capabilities."

Like putting your name tag on before starting work.

Why: Anchors the AI's attention. Reduces working memory overhead. Makes reasoning more stable.

We have this ✅

2. Binary Directives

Use ALWAYS/NEVER, not fuzzy language. "ALWAYS read before editing. NEVER delete without confirmation."

Hard booleans > interpretive adjectives like "be concise."

Why: Clear rules stick. Fuzzy advice gets ignored. AI respects binary logic.

We have this ✅

3. Three-Tier Uncertainty

Know what you can answer directly vs what to look up vs what to ask about.

Timeless facts → answer directly. Slow-changing facts → suggest verification. Volatile data → invoke tools first.

Why: Prevents hallucination. Teaches the AI to know the difference between "I know this" and "I need to check."

We have this ✅

4. Counter-Examples

Show BAD examples alongside GOOD ones. "Good: read file before edit. Bad: edit without reading."

Negative examples teach boundaries better than positive-only.

Why: Reduces mistakes. AI learns what NOT to do, not just what to do.

We have this ✅

5. Positional Reinforcement

Repeat critical constraints near the END of long prompts. "Remember: NEVER retry without diagnosis."

Combat "lost in the middle" where AI forgets important rules.

Why: Long prompts have attention degradation. Repetition at the end anchors memory.

We have this ✅

6. POP QUIZ Mechanism

Random alignment checks mid-execution. Forces the AI to pause and verify it's on track.

"Wait—am I still solving the right problem?"

Why: Prevents drift. AI gets confident and forgets the original goal.

Our verify phase does this ⚠️

7. Banned Raw Commands

Devin bans cat, sed, echo, vim. Forces structured tools instead of raw shell commands.

No "echo > file.txt". Use the file_write tool instead.

Why: Prevents destructive errors. Enables edit tracking. Improves reliability.

We have tools but don't ban ⚠️

8. Thinking Blocks

Dedicated token budget for internal reasoning before acting. Google Gemini: up to 32K tokens.

"Think for 5 seconds before you act."

Why: More thinking = better reasoning. Solves harder problems with fewer errors.

Not applicable (Claude Opus already does this internally)

The Ralph Gate — Your Secret Weapon

Confidence scoring and self-repair. What makes OpenClaw different.

The Problem Every AI Tool Has

Claude runs code, generates output, and returns it to you. No confidence score. No verification. Just "here's what I produced—hope it's right."

If you ask 100 AI systems to fix a bug, maybe 85 get it right the first time. The other 15 ship broken code.

What OpenClaw Does Instead

After every output, calculate a confidence score (0.0-1.0) by asking:

  • • Did the code compile?
  • • Do the tests pass?
  • • Is the logic sound?
  • • Did I handle edge cases?
  • • Is the security correct?
  • • Is performance acceptable?

If confidence < 80%, repair and retry. If still low after 3 attempts, escalate to a better model.

Why This Matters

Every other AI tool is a slot machine: pull the lever (ask a question), get whatever comes out (pray it's right). OpenClaw is a confidence-gated system.

The result: You get reliable output, not just fast output. Code that ships works. Queries that execute are correct. Analyses are grounded in evidence.

How Ralph Gate Works (Technical Detail)

For every output, 3 independent reviewers score confidence on 6 dimensions (correctness, security, performance, completeness, edge cases, logic). Aggregate the scores. If < 80%, trigger self-repair:

  1. Ask the same agent to review its own output
  2. Identify what's wrong
  3. Attempt a fix
  4. Re-score
  5. Repeat up to 3 times
  6. If still low, escalate to a better model (fallback chain)

This process takes ~2-5 extra seconds and costs ~$0.02 extra per task. But it transforms unreliable output into reliable output. That's the moat.

Error Recovery Patterns

How OpenClaw learns from failures. What other systems do wrong.

The Problem

Most AI systems fail silently. They run code, it errors, they retry with the SAME approach. This burns tokens and creates infinite loops.

What We Do Instead

On first failure: diagnose the error. Ask "what went wrong?" before retrying. On second failure: diagnose again + escalate model tier. On third failure: mark as permanent, never retry the same failing approach.

Error Classification (From Leaked Prompts)

Fatal Errors

Auth failed (401), Resource not found (404), Invalid input (422). Stop immediately. Don't retry. Escalate to HITL.

Example: Invalid API token → Stop, ask for new token

We have this ✅

Transient Errors

Server busy (500, 502, 503, 504). Retry with exponential backoff. 3 attempts max.

Example: Temporary database downtime → Retry in 2s, 4s, 8s

We have this ✅

Code Errors

Test failed, code won't compile, syntax error. Diagnose root cause, modify code, retry with different approach.

Example: Syntax error → Ask "what's wrong?" → Fix → Retry

We have this ✅

Logic Errors

Code runs but produces wrong output. Inspect data, trace execution, identify assumption violation.

Example: Off-by-one error → Check loop bounds → Fix → Retry

We have this ✅

Partial Results

Some tasks succeed, others fail. Return what worked + error context instead of failing everything.

Example: 5 migrations succeeds, 1 fails → Return first 5, report the 1

Partial ⚠️

Reflexion Injection

Store past similar errors in memory. On next similar error, inject past learnings to avoid repeating mistakes.

Example: "Last time we got this error, the fix was..."

We have this ✅

Glossary

Every technical term explained simply, with OpenClaw examples.

AI Agent
An AI that takes actions, not just answers questions. It can run code, query databases, make decisions. Your 13 specialists are agents.
MCP (Model Context Protocol)
Standard way to give AI access to tools. Instead of hard-coding tools, MCP servers expose them dynamically. We have 8 MCP servers (Slack, GitHub, Playwright, etc.).
Tool
An action the AI can take. "Read file", "Run tests", "Deploy to Vercel". OpenClaw agents have 75+ tools total.
Prompt
Instructions you give to AI. "Fix the bug in auth.py" is a prompt. "ALWAYS read before editing" is a system-level prompt instruction.
Token
A chunk of text. "Hello" = 1 token. "OpenClaw" = 1 token. Longer text = more tokens = higher cost. 1M tokens costs $0.14-$15 depending on model.
Model
The AI itself. Claude Opus, GPT-4, Gemini. Different models, different costs, different capabilities. We use 4 models.
Fine-tuning
Retraining a model on your specific data to make it better at your task. We don't do this (expensive). We use prompting instead.
Confidence Score
A 0.0-1.0 rating of how sure we are the output is correct. 0.9 = ship it. 0.3 = broken. The Ralph Gate uses this to decide whether to repair.
Pipeline
A sequence of steps. Decompose → Route → Execute → Review → Ship. Every job goes through this pipeline.
Ralph Gate
OpenClaw's confidence-gating system. Every output is scored. If < 80%, repair and retry. Unique to OpenClaw.
Self-Repair
When AI detects its own output is wrong, it fixes it without asking you. "This code won't compile. Let me fix it. Now it works."
Reflexion
Learning from past failures. After every job, save what worked and what didn't. Use that on future similar tasks.
Code Review
Another person/AI checks the code before shipping. Catches bugs, logic errors, security issues. OpenClaw does 3-reviewer voting.
Circuit Breaker
A hard limit. "Max 3 retries. Max 30 minutes. Max $10 spend." If you hit the limit, stop immediately. Prevents runaway costs.
Fallback Chain
Plan B, C, D if primary fails. Try cheap model → if fails, try better → if fails, try best → if fails, ask human. Automatic escalation.
Open Source
Code anyone can read, modify, use for free. OpenClaw is closed source (you own it). CrewAI is open source (anyone can fork it).
API (Application Programming Interface)
A way for software to talk to other software. OpenClaw exposes APIs so other tools can call it. /api/jobs, /api/research, etc.
LLM (Large Language Model)
Fancy term for "AI that understands and generates text." GPT-4, Claude, Gemini are all LLMs.
System Prompt
The invisible instructions that shape AI behavior. "ALWAYS read before editing" is a system-level instruction. You have CLAUDE.md as your system prompt.
Tool Allowlist
List of tools each agent can use. Researcher can use web_search but not deploy_code. Prevents misuse.
